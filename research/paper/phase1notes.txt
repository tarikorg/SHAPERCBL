What the diagnostics prove (paper-ready interpretation)
1) Small dataset warnings (4 total, all “rounding”)

From your diagnostics:

    SALARY values like 68000.9 and 79999.99 produce rounding warnings because JS floating-point scaling gives values like:
        68000.9 * 100 = 6800089.999999999
        79999.99 * 100 = 7999999.000000001

This is not a data problem—it’s a JavaScript floating point representation issue.

Research implication: If you leave this as-is, your warning counts may be inflated by implementation artifacts (threat to validity). We should fix it so warnings represent real rounding (e.g., input had too many decimal places), not IEEE-754 noise.
2) Large dataset warnings (200 total, all “rounding”)

Same root cause. LAST_RAISE values such as 2.55 lead to:

    2.55 * 100 = 254.99999999999997

So the pipeline warns about rounding even when the input is already “2 decimal places”. Again, this is IEEE-754.
